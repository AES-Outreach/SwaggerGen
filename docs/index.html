Introducing SwaggerGen, a free
tool to document your APIs at the code level

  • How it works

Problem Statement

Documenting APIs is an unavoidable evil to ensure that the service will be properly used and maintained. However, the process of sitting down to write the associated documentation may be easily overlooked as it remains a non-trivial task to accomplish, and requires a considerable time investment. This often leads unmaintained documentation as the code inevitably evolves, and the task of documentation is often relegated to the bottom of the priorities list in favour of others tasks that might be judged more urgent. This process exposes a major flaw of the standard documentation process; namely its reliance on the developers to set time apart from writing code in order to write or update the associated documentation.

What other solutions offer

Being based on Swagger (OpenAPI), there are some tools that inevitably share some similarities with SwaggerGen. One of them, swagger-maven-plugin uses Spring annotations to generate swagger specs during the maven build phase. It uses a heavy config file, which is something that we try to avoid. The is also Swagger-Core which is offered by the Swagger team. However, we found that the downsides to using this tool were its steep learning curve in understanding Swagger syntax and its many annotations. This is why we wanted to offer a tool that would be easy to use for even those unfamiliar with the internals of Swagger.

What makes us different

SwaggerGen is a project that aims to seamlessly integrate API documentation directly within the software development process. It does so by leveraging custom Java annotations that document the information of server endpoints at the method and class-level. The plugin uses the annotated information to automatically generate a Swagger (OpenAPI) file describing the functionalities of the RESTful services. This file can, in turn, be used with other tools such as SoapUI or Postman to obtain a visual representation of such services.
